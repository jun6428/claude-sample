# ADR 001: 資源カードをエンティティとしてモデリングする

## 決定

資源をプレイヤーごとの枚数カウント `Dict[str, int]` ではなく、
`ResourceCard` オブジェクトのリストで管理する。各カードは `holder`（"bank" or "player_N"）を持つ。

## 背景

資源カードは各種19枚という物理的な上限がある。
これは**通貨（無限に発行できる）ではなくコモディティ（有限な実物）**である。

## 理由

- 銀行在庫が枯渇した資源は配れない（`bank_stock()` で在庫チェック）
- ダイスロール時の分配は `distribute_from_bank` で「在庫不足なら誰にも配らない（all-or-nothing）」を保証できる
- カードの総枚数は常に一定（銀行＋全プレイヤーの合計 = 19枚）であり、エンティティモデルがこの不変条件を自然に表現する

### 「在庫チェック不要では？」という議論を根絶する

カウント方式では操作ごとに「この場合は在庫チェック要る？」を個別判断する必要が生じる。
例：
- セットアップ2ターン目の資源付与 → 不要？
- 盗賊による資源移動 → 不要？
- 港交易 → 要る？

エンティティ方式では**カードは必ずどこかの `holder` に属しており、移動するだけ**。
銀行在庫は `bank_stock()` の結果として自動的に正しい値になるため、
「チェックが必要かどうか」という問い自体が発生しない。

## 却下した設計

プレイヤーごとに `resources: Dict[str, int]`、銀行に `bank: Dict[str, int]` を持つシンプルなカウント方式。
この設計では銀行在庫チェックを別途実装する必要があり、不変条件の保証が弱くなる。

## 適用範囲

このパターンはゲーム内のすべての有限コンポーネントに適用している：

| コンポーネント | 上限 | エンティティ |
|--------------|------|------------|
| 資源カード | 各19枚 | `ResourceCard` |
| 道路コマ | 各15本 | `RoadPiece` |
| 開拓地コマ | 各5個 | `SettlementPiece` |
| 都市コマ | 各4個 | `CityPiece` |
| 発展カード | 合計25枚 | `DevCard`（実装予定） |

`supply_piece()` でコマ在庫を取得し、なければエラーにするだけでよい。
「個数制限のチェックを実装したか」「し忘れていないか」という議論が不要になる。

## 影響

- `state.py`: `ResourceCard`, `transfer_from_bank`, `transfer_to_bank`, `transfer_between_players` で一貫した操作
- 発展カードも同様に `DevCard` エンティティとしてモデリングすべき（デッキ枚数に上限あり）
